# Пользовательские сценарии для e2e

Сценарии ниже соответствуют [`booking.spec.ts`](booking.spec.ts) и ручной проверке. Даты в автотестах всегда в **будущем** относительно времени сервера (иначе API вернёт 422).

| Сценарий | Предусловия | Шаги | Ожидание |
|----------|-------------|------|----------|
| **Основной: бронирование** | Чистый запуск API (в памяти нет типов встреч) | `/owner` → заполнить id, название, описание, длительность → «Создать тип» → `/` → выбран тип, **будущий** диапазон дат → «Показать свободные слоты» → выбрать слот → имя, email → «Забронировать» | Уведомление «Бронирование создано»; на `/owner` после «Обновить список» видна строка с гостем |
| **Редактирование типа** (опционально, не в минимальном e2e) | Есть тип встречи | `/owner` → «Открыть» у типа → изменить поля → «Сохранить» | Уведомление «Сохранено» |

Запуск тестов: из корня `mise run e2e:test` или как в CI: `mise run e2e:ci`. См. [`README.md`](../README.md).
